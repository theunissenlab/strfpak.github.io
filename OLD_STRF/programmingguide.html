Programming Guide

Purpose:

Your code should be easy to read and interpret. This is especially important at a later time if you or someone else updates your code. The following guides will help you develop a programming style that will enhance the readability of your code:

Comment:

It is considered good programming practice to include comment lines. Comment lines, indicated by a % in Matlab or /* xxx */ in C, add information, such as the purpose of the program, definition of variables, and the purpose of statements within the program. A comment indicating the units of measure that correspond to numeric values will also help in the interpretation of results. Comments are printed in the listing of a program, but are otherwise ignored; they do not affect program execution. It is recommended that each code be commented at the beginning to tell

  • What it does;
  • How it is being done;
  • What the parameters mean;
  • Who wrote the code and when;
  • Revision History;
 

Blank lines:

Within the code, use blank lines to separate the different steps of the algorithm.

Modularization:

A fundamental approach to solving large, complex problems is to break the problem down into smaller problems. Then, solutions can be found for these smaller problems and organized in such a manner as to address the original problem. In a similar way, problems that require long, complex Matlab code can be broken down into function units. In addition to the main program unit, Matlab or C programs can call Matlab function or subprogram written in other languages. Modularization allow the programmer to organize programs in a logical, hierarchical fashion. There are several advantages to modularize your program:

  • they help keep the code simple, and, thus, more readable;
  • they allow the programmer to use the same code as many times as needed throughout the program;
  • they allow the programmer to define needed functions; and,
  • they can be used in other programs.


Example
Assume that this is the pseudo-code for a desired sequence of operations:

read the data from a file;
perform calculations based on the data read;
post-process the data; and,
save the data in a new file.


The program could be written with five program units: a main program and four subprograms: GETFILE, CALCULATION, PPROCESS and NEWFILE. This will help to add or modify some modules later and improve the code's maintainability.
 

Documentation:

It is strongly recommended that every application or run procedure have a README file which documents the following things:

  • General description of the application.
  • Description of what the code does
  • Who wrote the code and when.
  • Algorithms used.
  • Library and program dependencies.
  • Design restriction.
  • Inputs.
  • Output.
  • Known bugs.
  • Revision History.
